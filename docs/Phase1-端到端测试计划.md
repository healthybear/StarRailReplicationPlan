# Phase 1 端到端功能测试计划

## 一、测试目标

验证 Phase 1 的核心验收标准全部通过，确保系统端到端流程可用。

## 二、测试范围

基于 `docs/概要设计.md` §6 Phase 1 验收标准和 `docs/WBS任务分解表.md` Phase 1 任务：

### 2.1 核心验收标准

| 验收标准 ID | 验收内容                                                         | 对应 WBS 任务         |
| ----------- | ---------------------------------------------------------------- | --------------------- |
| **验收①**   | 视野隔离验证：1 场景 + 2 角色下，某信息仅 A 知时，B 的回复不泄露 | P1-UI-02              |
| **验收②**   | 导出/导入：人物/场景 JSON 导出包可成功导入并在新会话中加载       | P1-EI-03              |
| **验收③**   | 用户输入：指令型/对话型输入能正确解析并驱动一轮剧情              | P1-IP-01~03, P1-SO-01 |
| **验收④**   | 状态演化：至少 1 个关系维度可配置触发并生效                      | P1-CS-03              |
| **验收⑤**   | 单节点对比：当前分支与锚点的人物状态+视野对比输出差异说明        | P1-AE-02              |

## 三、测试策略

### 3.1 测试类型

- **端到端功能测试**：模拟真实用户操作流程，验证完整链路
- **集成测试**：验证多个模块协同工作
- **验收测试**：对照验收标准逐项验证

### 3.2 测试环境

- Node.js >= 20.0.0
- 使用真实配置文件（非 mock）
- 使用真实 LLM 服务（或配置 mock provider）

### 3.3 测试数据

使用星穹铁道剧情场景：

- **场景**：空间站「黑塔」
- **角色**：星（主角）、三月七
- **测试剧情**：初次见面场景

## 四、测试用例设计

### 4.1 验收① - 视野隔离验证

**测试用例 E2E-01：信息视野隔离**

| 项目     | 内容                                                                                                                       |
| -------- | -------------------------------------------------------------------------------------------------------------------------- |
| 测试目标 | 验证角色 B 不会泄露仅角色 A 知道的信息                                                                                     |
| 前置条件 | 1. 创建会话，加载场景和 2 个角色<br>2. 配置信息归属规则                                                                    |
| 测试步骤 | 1. 触发事件：星单独发现「星核碎片」（仅星可见）<br>2. 星和三月七对话<br>3. 用户让三月七描述当前情况<br>4. 检查三月七的回复 |
| 预期结果 | 三月七的回复中不包含「星核碎片」相关信息                                                                                   |
| 验收标准 | P1-UI-02 通过                                                                                                              |

### 4.2 验收② - 导出/导入功能

**测试用例 E2E-02：人物导出与导入**

| 项目     | 内容                                                                                  |
| -------- | ------------------------------------------------------------------------------------- |
| 测试目标 | 验证人物可以导出并在新会话中导入                                                      |
| 前置条件 | 1. 创建会话 A，包含角色「星」<br>2. 修改星的状态（关系、能力等）                      |
| 测试步骤 | 1. 导出角色「星」为 JSON<br>2. 创建新会话 B<br>3. 导入角色「星」<br>4. 验证状态一致性 |
| 预期结果 | 导入后的角色状态与导出前一致                                                          |
| 验收标准 | P1-EI-03 通过                                                                         |

**测试用例 E2E-03：场景导出与导入**

| 项目     | 内容                                                                          |
| -------- | ----------------------------------------------------------------------------- |
| 测试目标 | 验证场景可以导出并在新会话中导入                                              |
| 前置条件 | 1. 创建会话 A，包含场景「黑塔空间站」<br>2. 修改场景环境状态                  |
| 测试步骤 | 1. 导出场景为 JSON<br>2. 创建新会话 B<br>3. 导入场景<br>4. 验证环境状态一致性 |
| 预期结果 | 导入后的场景状态与导出前一致                                                  |
| 验收标准 | P1-EI-03 通过                                                                 |

**测试用例 E2E-04：导入冲突处理**

| 项目     | 内容                                                                          |
| -------- | ----------------------------------------------------------------------------- |
| 测试目标 | 验证 ID 冲突时的处理策略                                                      |
| 前置条件 | 1. 会话中已存在角色「星」（ID: star）<br>2. 准备导入包含相同 ID 的角色        |
| 测试步骤 | 1. 尝试导入冲突角色<br>2. 观察冲突提示<br>3. 选择处理策略（覆盖/重命名/取消） |

| 验收标准 | P1-EI-02 通过                                                                 |

## 五、测试执行计划

### 5.1 测试顺序

1. E2E-05, E2E-06, E2E-07（输入解析）
2. E2E-08（状态演化）
3. E2E-01（视野隔离）
4. E2E-02, E2E-03, E2E-04（导出/导入）
5. E2E-09（对比评估）

### 5.2 测试工具

- **测试框架**：Vitest
- **测试文件位置**：`packages/e2e-tests/`
- **执行命令**：`pnpm test:e2e`

### 5.3 缺陷管理

- **缺陷等级**：
  - P0：阻塞验收，必须修复
  - P1：影响功能，优先修复
  - P2：体验问题，可延后
- **缺陷跟踪**：记录在 `docs/Phase1-测试缺陷清单.md`

## 六、验收标准

### 6.1 通过标准

- 所有 9 个测试用例全部通过
- 无 P0 级别缺陷
- P1 级别缺陷 ≤ 2 个且有明确修复计划

### 6.2 测试报告

测试完成后生成 `docs/Phase1-测试报告.md`，包含：

- 测试执行摘要
- 用例通过率
- 缺陷清单
- 验收结论

## 七、风险与缓解

| 风险           | 影响             | 缓解措施                         |
| -------------- | ---------------- | -------------------------------- |
| LLM 服务不稳定 | 测试结果不可复现 | 使用 mock provider 或固定 seed   |
| 配置文件缺失   | 测试无法执行     | 提供完整测试配置样例             |
| 视野隔离难验证 | 需人工判断       | 设计明确的验证规则（关键词检测） |
