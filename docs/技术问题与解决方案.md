## 一、模块系统兼容性问题

### 问题描述

**问题**: NestJS API 服务（使用 CommonJS）无法正确导入 Core 包（使用 ES Modules）

**错误信息**:

```
Error [ERR_REQUIRE_ESM]: require() of ES Module not supported
Error [ERR_PACKAGE_PATH_NOT_EXPORTED]: No "exports" main defined
```

**根本原因**:

1. Core、Infrastructure、Types 包的 `package.json` 中设置了 `"type": "module"`
2. 这些包的 `exports` 字段只定义了 `"import"`，没有 `"require"`
3. NestJS 默认使用 CommonJS 模块系统
4. TypeScript 编译配置不一致（有的用 ESNext，有的用 CommonJS）

### 解决方案

**步骤 1**: 统一使用 CommonJS 模块系统

在 `packages/core/package.json`、`packages/infrastructure/package.json`、`packages/types/package.json` 中：

```json
{
  // 移除这一行
  // "type": "module",

  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.js" // 添加 require 入口
    }
  }
}
```

**步骤 2**: 更新 TypeScript 配置

在 `packages/core/tsconfig.json` 中：

```json
{
  "compilerOptions": {
    "module": "CommonJS", // 改为 CommonJS
    "moduleResolution": "node"
  }
}
```

**步骤 3**: 更新构建脚本

在 `packages/infrastructure/package.json` 和 `packages/types/package.json` 中：

```json
{
  "scripts": {
    "build": "tsup src/index.ts --format cjs --dts", // 改为 cjs
    "dev": "tsup src/index.ts --format cjs --dts --watch"
  }
}
```

### 最佳实践

1. **Monorepo 中保持模块系统一致性**: 所有包应使用相同的模块系统（CommonJS 或 ES Modules）
2. **package.json exports 字段**: 如果需要同时支持 CommonJS 和 ES Modules，必须同时提供 `"require"` 和 `"import"` 入口
3. **TypeScript 配置对齐**: 确保 `tsconfig.json` 的 `module` 设置与 `package.json` 的 `type` 字段一致

### 相关文件

- `packages/core/package.json`
- `packages/core/tsconfig.json`
- `packages/infrastructure/package.json`
- `packages/infrastructure/tsconfig.json`
- `packages/types/package.json`

### 参考资料

- [Node.js ES Modules 文档](https://nodejs.org/api/esm.html)
- [TypeScript Module Resolution](https://www.typescriptlang.org/docs/handbook/module-resolution.html)
- [Package.json exports 字段](https://nodejs.org/api/packages.html#exports)

## 二、依赖注入容器桥接问题

### 问题描述

**问题**: NestJS 使用自己的 DI 容器，Core 包使用 tsyringe DI 容器，两者无法直接互通

**错误信息**:

```
Error: Attempted to resolve unregistered dependency token: "StoryOrchestrator"
Cannot inject dependencies - TypeInfo not known for "String"
```

**根本原因**:

1. 使用字符串 token（如 `'StoryOrchestrator'`）而非类 token
2. NestJS 和 tsyringe 是两个独立的 DI 系统
3. Core 服务在 tsyringe 容器中注册，但 NestJS 无法直接访问

### 解决方案

**步骤 1**: 创建 Provider 工厂桥接两个 DI 系统

创建 `packages/api/src/common/providers/core.provider.ts`:

```typescript
import { container } from 'tsyringe';
import { Provider } from '@nestjs/common';
import {
  StoryOrchestrator,
  SnapshotService,
  // ... 其他服务
} from '@star-rail/core';

// 定义 token 常量
export const STORY_ORCHESTRATOR = 'STORY_ORCHESTRATOR';
export const SNAPSHOT_SERVICE = 'SNAPSHOT_SERVICE';

export function createCoreProviders(configService: ConfigService): Provider[] {
  // 初始化 tsyringe 容器
  initializeContainer(configService);

  return [
    {
      provide: STORY_ORCHESTRATOR,
      useFactory: () => container.resolve(StoryOrchestrator), // 使用类 token
    },
    {
      provide: SNAPSHOT_SERVICE,
      useFactory: () => container.resolve(SnapshotService),
    },
    // ... 其他服务
  ];
}
```

**关键点**:

- 使用 `container.resolve(StoryOrchestrator)` 而非 `container.resolve('StoryOrchestrator')`
- 导入实际的类作为 token，而非字符串

**步骤 2**: 创建全局 CoreModule

创建 `packages/api/src/common/modules/core.module.ts`:

```typescript
import { Global, Module } from '@nestjs/common';
import { ConfigService } from '@nestjs/config';
import { createCoreProviders } from '../providers/core.provider';

@Global() // 标记为全局模块
@Module({
  providers: createCoreProviders(new ConfigService()),
  exports: createCoreProviders(new ConfigService()).map((p) =>
    typeof p === 'object' && 'provide' in p ? p.provide : p
  ),
})
export class CoreModule {}
```

**步骤 3**: 在 AppModule 中导入 CoreModule

```typescript
@Module({
  imports: [
    ConfigModule.forRoot({ isGlobal: true }),
    CoreModule, // 导入全局模块
    SessionModule,
    StoryModule,
    // ...
  ],
})
export class AppModule {}
```

**步骤 4**: 在业务模块中注入服务

```typescript
@Injectable()
export class SessionService {
  constructor(
    @Inject(STORY_ORCHESTRATOR)
    private readonly storyOrchestrator: StoryOrchestrator
  ) {}
}
```

### 最佳实践

1. **使用类 token 而非字符串**: tsyringe 支持类作为 token，更类型安全
2. **创建全局模块**: 使用 `@Global()` 装饰器避免在每个模块中重复导入
3. **Provider 工厂模式**: 通过工厂函数桥接不同的 DI 系统
4. **统一初始化**: 在 Provider 工厂中统一初始化 tsyringe 容器

### 相关文件

- `packages/api/src/common/providers/core.provider.ts`
- `packages/api/src/common/modules/core.module.ts`
- `packages/api/src/app.module.ts`

### 参考资料

- [NestJS Custom Providers](https://docs.nestjs.com/fundamentals/custom-providers)
- [NestJS Global Modules](https://docs.nestjs.com/modules#global-modules)
- [tsyringe Documentation](https://github.com/microsoft/tsyringe)

## 三、TypeScript 配置最佳实践

### Monorepo 包配置规范

**根目录 tsconfig.json** (基础配置):

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["ES2022"],
    "module": "CommonJS",
    "moduleResolution": "node",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  }
}
```

**包级别 tsconfig.json** (继承并扩展):

```json
{
  "extends": "../../tsconfig.json",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src",
    "noEmit": false,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true
  },
  "include": ["src/**/*"],
  "exclude": ["src/**/__tests__/**", "src/**/*.test.ts"]
}
```

### 关键配置说明

| 配置项                   | 说明            | 推荐值                       |
| ------------------------ | --------------- | ---------------------------- |
| `module`                 | 模块系统        | `CommonJS`（与 NestJS 兼容） |
| `moduleResolution`       | 模块解析策略    | `node`                       |
| `experimentalDecorators` | 装饰器支持      | `true`（NestJS 必需）        |
| `emitDecoratorMetadata`  | 装饰器元数据    | `true`（DI 必需）            |
| `declaration`            | 生成 .d.ts      | `true`（库包必需）           |
| `sourceMap`              | 生成 source map | `true`（调试友好）           |

## 四、ESLint 配置问题

### 测试文件中的类型断言

**问题**: 测试文件中使用 `as any` 会触发 ESLint 错误

**解决方案**: 在 `eslint.config.mjs` 中为测试文件添加规则覆盖

```javascript
export default [
  // ... 其他配置
  {
    files: ['**/*.spec.ts', '**/*.test.ts'],
    rules: {
      '@typescript-eslint/unbound-method': 'off',
      '@typescript-eslint/no-unsafe-assignment': 'off',
    },
  },
];
```

### 最佳实践

1. **创建类型化的 mock 对象**: 避免使用 `as any`
2. **测试文件规则放宽**: 测试代码可以适当放宽类型检查
3. **使用 Jest 类型**: 利用 `jest.Mock<T>` 等类型工具

## 五、常见错误速查表

| 错误信息                        | 可能原因                  | 解决方案                          |
| ------------------------------- | ------------------------- | --------------------------------- |
| `ERR_REQUIRE_ESM`               | 尝试 require ES Module    | 改用 CommonJS 或添加 require 入口 |
| `ERR_PACKAGE_PATH_NOT_EXPORTED` | exports 字段缺少入口      | 添加 "require" 或 "import" 入口   |
| `Unregistered dependency token` | DI token 未注册或类型错误 | 使用类 token 而非字符串           |
| `Cannot inject dependencies`    | 构造函数参数类型问题      | 检查装饰器和参数类型              |
| `Module not found`              | 路径或包名错误            | 检查 package.json name 和 imports |

## 六、开发环境配置检查清单

开始新模块开发前，检查以下配置：

```
□ package.json 的 "type" 字段与项目一致
□ package.json 的 "exports" 包含所需的入口（require/import）
□ tsconfig.json 的 "module" 与 package.json 一致
□ 构建脚本的 --format 参数正确（cjs/esm）
□ 依赖包已正确安装（pnpm install）
□ TypeScript 编译无错误（pnpm build）
□ ESLint 检查通过（pnpm lint）
□ 测试通过（pnpm test）
```
